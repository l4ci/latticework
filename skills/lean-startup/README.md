# lean-startup

## What this is

Lean Startup is a method for building products and businesses under extreme uncertainty by maximizing validated learning per unit of time and money. Instead of writing a big plan and executing it, the venture is treated as a set of untested hypotheses, and the smallest possible experiment is run against the riskiest one. Progress is measured as validated learning about customers, not features shipped, and each cycle ends in a deliberate decision to pivot or persevere. The method was set out by Eric Ries in *The Lean Startup* (2011), drawing on lean manufacturing and Steve Blank's customer development work.

The core concepts (leap-of-faith assumptions, the Build-Measure-Learn loop, the MVP, actionable versus vanity metrics, and pivot or persevere) are in [references/lean-startup.md](references/lean-startup.md).

## What it does

The skill runs one Build-Measure-Learn loop end to end and sets up the next. It helps surface the leap-of-faith assumptions (including the value and growth hypotheses) as falsifiable statements, then ranks them by risk using parallel analysts working different lenses (most uncertain, most fatal if false, what a skeptical investor would attack), so the first experiment tests the riskiest assumption rather than the easiest. It then designs the minimum viable test with an actionable metric and a pass-and-fail threshold committed in advance, captures what customers actually did, and recommends pivot or persevere on that evidence. The output is a saved markdown experiment document with a loop log. Three guardrails keep it honest: the MVP is a learning tool rather than a small product, vanity metrics that rise but test nothing are rejected, and the threshold is set before running so results cannot be rationalized after the fact.

## When to use it

Reach for this skill when a product or business idea faces real uncertainty about whether customers want it or whether it will grow, and you want to test it cheaply before committing.

- If the question is designing the solution itself around user needs, use [design-thinking](../design-thinking/) upstream; this skill validates that the thing should exist.
- If you need to understand the underlying job and why customers would switch, use [jobs-to-be-done](../jobs-to-be-done/).
- If you want to map the whole business model and its viability, not just one hypothesis, use [business-model-canvas](../business-model-canvas/).
