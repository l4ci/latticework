# design-thinking

## What this is

Design Thinking is a human-centered approach to innovation: understand the people you are designing for, frame the right problem, generate many options, make ideas tangible cheaply, and learn from real feedback. The version used here follows the five-mode framing popularized by the Stanford d.school and IDEO in the 2000s: Empathize, Define, Ideate, Prototype, Test. These are modes of work, not strict sequential steps. The method runs as a loop: an insight from a test can send the work back to reframing the problem or back to talking to users, and that looping is the method working as intended.

The five modes, how to use them, and the pitfalls are in [references/design-thinking.md](references/design-thinking.md).

## What it does

The skill facilitates the work one mode at a time and keeps a living design document. It leads with real user needs, since the main way design work drifts is toward the solution and away from the user. During Ideate it diverges widely before converging: several idea-generator subagents run in parallel, each using a different ideation lens (analogous domains, extreme users, constraint removal, and so on), so the pooled ideas are varied rather than ten versions of one, before the user picks a few to prototype. Prototypes are kept deliberately cheap and rough, and testing is framed to seek disconfirming feedback, not applause. Each loop ends by deciding which mode the test results send the work back to, and the design document records the modes, the ideas, the prototypes, and what each test disconfirmed.

## When to use it

Reach for this skill to design or redesign a product, service, feature, or experience around real user needs, especially when the right problem is not yet clear.

- If the problem is already clear and you only need to sharpen it into a crisp statement, use [tosca-problem-definition](../tosca-problem-definition/).
- If the question is whether the thing should exist and will work, test it cheaply with [lean-startup](../lean-startup/); a Design Thinking concept can feed that experiment.
- If you need to understand the underlying job a customer is trying to get done, use [jobs-to-be-done](../jobs-to-be-done/) before ideating.
